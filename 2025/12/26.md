## わかったこと
- `SELECT 1 ... LIMIT 1` は存在チェック専用のSQLでDjangoの `QuerySet.exists()` が内部的に同じことをしている
- QuerySet は「取得済みデータ」ではなく「取得ルール」であり`list()` にした瞬間にDBアクセスして実体が確定する
- 注文確定のような業務処理ではQuerySetを `list()` 化してスナップショットを取る意味がある
- `form.data` は Form を `request.POST` で生成した瞬間に自動で作られる「生データ置き場」
- `form.cleaned_data` は `is_valid()` 後に確定する「検証済みデータ」で役割が明確に違う
- Djangoテンプレートの `|default_if_none` や自作フィルタはどちらも「値を加工・差し替えするフィルタ」という同じ仕組み
- フォームバリデーションは、required → 型/形式 → `clean_<field>` → `clean()` の順で進む
- `with` は「必ず後始末するための構文」で`transaction.atomic()` はDB操作をまとめるためのもの
- `ModelForm` は「フォーム生成 → バリデーション → save」で初めてモデルインスタンスを作る
- `save(commit=False)` は「まだ保存しないモデルインスタンス」を作るための重要な仕組み
- Djangoでは URL は画面ではなく「処理の入口」であり、画面があるかは View の戻り値次第
- `default` フィルタは加工ではなく「フォールバック（空なら次）」の役割を持つ特殊なフィルタ
